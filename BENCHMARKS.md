# Benchmarks

This file is not optional decoration — it's the single highest-leverage piece of
credibility evidence in this whole project. A claim like "uses spatial indexing
for efficient queries" is worth nothing in an interview. A real number is worth
more than every other line on your resume bullet for this project combined.

## Spatial query: naive filter vs. PostGIS + GiST index

**Environment:** local Docker `postgis/postgis:16-3.4` container, Windows host,
query for `/aqi/current?lat=47.6062&lon=-122.3321&radius_miles=25` (Seattle).

**Data:** 3,016 stations / 3,053 readings total. 16 stations / 53 readings are
real, from ingesting 16 actual US cities through the AirNow API. The
remaining 3,000 stations / 3,000 readings are **synthetic** — real AirNow
station coverage across the entire US is only a few hundred to ~1,500
stations, nowhere near enough to show an index effect at scale. Synthetic
rows are generated by `scripts/seed_benchmark_data.py` (continental US
bounding box, random lat/lon/pollutant/AQI) and are clearly distinguishable
by an `external_id` prefixed `SYNTHETIC-`. This disclosure is here because a
benchmark that hides synthetic data behind "real-looking" numbers is worse
than no benchmark at all.

**Two queries compared**, both against the same 3,016-row `stations` table:
- **Naive**: `db.query(Station).all()`, then a manual lat/lon bounding-box
  filter in Python (`app/api/main.py::_naive_nearby_stations`)
- **Indexed**: a `&&` bounding-box check directly on the GiST-indexed
  `location` column, narrowing candidates before an exact
  `ST_DWithin(location::geography, point::geography, meters)` filter
  (`app/api/main.py::_indexed_nearby_stations`) — see the comment above
  `INDEXED_NEARBY_STATIONS_SQL` for why the query has this shape: an
  earlier version that only cast to `::geography` (closer to a naive
  reading of the plan below) turned out to force a full sequential scan,
  verified via `EXPLAIN ANALYZE`, not assumed.

### Raw SQL execution time (`EXPLAIN ANALYZE`, Postgres-side only)

| Approach | Row count | Execution time | Query plan |
|---|---|---|---|
| Naive (`SELECT * FROM stations`) | 3,016 | 0.42 ms | `Seq Scan on stations` |
| PostGIS + GiST index | 3,016 | 12.40 ms | `Index Scan using idx_stations_location` |

**This result is real and it's the opposite of the headline claim** — at
this table size, the sequential scan is faster than the index scan. This
isn't a mistake: GiST index traversal and the per-row geography distance
check both carry real overhead, and a 3,016-row table is small enough to
scan sequentially faster than an index can narrow it down. This is the
honest, well-known behavior of query planners at small scale — indexes
aren't free, and it's a real engineering judgment call to know when a table
is big enough for one to pay off.

### Full API round-trip time (`time.perf_counter()` around real HTTP requests)

30 requests per method against a live local `uvicorn` server (not
`TestClient` — real HTTP, to capture the full path):

| Approach | Mean | Median | Min | Max |
|---|---|---|---|---|
| Naive (`?method=naive`) | 46.89 ms | 40.80 ms | 38.07 ms | 83.27 ms |
| Indexed (`?method=indexed`) | 9.01 ms | 8.94 ms | 7.79 ms | 11.03 ms |

**The indexed approach is ~5x faster end-to-end**, reversing the raw-SQL
picture above. The reason: `EXPLAIN ANALYZE` only measures time inside
Postgres. The naive path's `db.query(Station).all()` ORM-hydrates **all**
3,016 rows into full Python objects before the bounding-box filter discards
nearly all of them — that hydration cost dominates the real request time
and doesn't show up in Postgres's own query plan at all. The indexed path
only ever hydrates the handful of rows that actually match.

**Conclusion:** the claim "PostGIS + a spatial index beats a naive filter"
is true where it actually matters — the real, measured, end-to-end request
time a user or another service would experience — but the *reason* isn't
"the index makes the SQL query faster" (at this row count it measurably
doesn't). It's that pushing the filter into the database avoids paying
Python-side ORM overhead for rows you're about to throw away. That's a more
precise and more defensible claim than the generic version, and it's the
one worth being able to explain in an interview.
